1 Introduction

CausalTrail is a tool for causal hypotheses testing. It allows the formulation of three types of queries:

  • Predictions,
  • Interventions,
  • Counterfactuals.

The graphical user interface is intended to simplify the usage of our tool. This documentation introduces our graphical user interface and details how it can be used.

2 Build CausalTrail including the Graphical User Interface

The following description is valid for linux systems only. We use cmake to build CausalTrail. There are two prerequisites that must be fulfilled to build our tool:

  1. The boost library, version 1.55.0 must be installed
  2. A C++ compiler supporting C++11 must be installed

To build CausalTrail, execute the following steps:

  1. Go to the CausalTrail directory
  2. Create a build folder, e.g. by typing mkdir build in the console
  3. Enter the build folder
  4. Execute cmake via cmake ..
  5. Build the project by typing make
  6. To build the tests of CausalTrail enter the path to gtest with the command cmake . -DGTEST_SRC_DIR=<Path to gtest>
  7. Set the path to Qt by enter cmake . -DQt5Widgets_DIR=<Path to Qt>
  8. Build the project by typing make

The executable of the gui is located in the folder build/gui, the consol version is located in the folder build/core, and tests can be found in the folder build/test.

3 Supported Data Formats

3.1 Network Formats

CausalTrail supports the following network formats:

  • trivial graph format (tgf),
  • simple interaction format (sif) and node attribute (na) files.

3.1.1 Trivial Graph Format

The Trivial Graph Format (TGF) has the following structure:

NodeID NodeAttribute
...
#
NodeID NodeID EdgeAttribute
...

The upper part of a TGF file, contains the mapping between node identifiers and at most one optional attribute, e.g. node names. The # marks the beginning of the actual network definition. Edges are directed from the first to the second node identifier. An edge between two nodes can be mapped to at most one optional attribute.

3.1.2 Node Attribute Format

Node Attribute files (NA) have the following structure:

AttributeName (class = Type)
NodeID = NodeAttribute
...

Different node attribute classes can be referenced via the AttributeName. Type states the data type of the node attributes in terms of a Java class. The mapping of NodeID to NodeAttribute has to be unique within one class of attributes.

3.1.3 Simple Interaction Format

The Simple Interaction Format (SIF) has the following structure:

...
NodeID EdgeType NodeID
...

Nodes in the network are identified via the NodeID. Therefore they have to be unique. The left NodeID represents the source of an edge, the right one represents the target. It is possible to assign more than one target node to a single source node, so multiple edges can be encoded in one line. The Edge Type encodes the type of an edge, e.g. whether an edge between two nodes is directed or not. It is also common to encode biological meaning in the Edge Type. For example, pd represents Protein-DNA interactions, whereas pp represents Protein-Protein interactions. The Edge Type can also be longer string. This allows the encoding of more complex descriptions, e.g. activates, inactivates, or phosphorylates. If it is not necessary to encode any specific meaning for an edge, xx or yy can be used as an Edge Type.

3.2 Data

3.3 Discretisation Information

The discretisation control file is a regular text file. Its rows are tab-delimited and structured as follows:

RowIndex DiscretisationMethod OptionalValue

The RowIndex is the number of the row in the sample file upon which the specified discretisation method should be applied.
The DiscretisationMethod is an integer coding for one of the discretisation methods included in CausalTrail. The coding obeys the following system:

  • 0 → Round to the next largest integer value.
  • 1 → Round to the next smallest integer value.
  • 2 → Round to the nearest integer value.
  • 3 → Thresholding according to the arithmetic mean.
  • 4 → Thresholding according to the harmonic mean.
  • 5 → Thresholding according to the median.
  • 6 → Thresholding according to a manually set threshold.
  • 7 → Use the Bracket Median method.
  • 8 → Use the Pearson Tukey method.
  • 9 → The data is discrete.
  • 10 → Thresholding according to the Z-score.

The OptionalValue needs to be specified for methods 6 and 7. Using our GUI, the discretisation control file can be created interactively.

To simplify the usage of our tool and to avoid repeating the process of network and sample loading, CausalTrail supports sessions. The dialogue shown in the left part of Figure 11 appears. The user has to specify a CausalTrail Session File (cts). To load a stored session, choose Load Session in the toolbar or click File → Load Session in the menu. The right part of Figure 11 shows the dialogue to select a cts file. Loaded networks will be added to the current session of CausalTrail.


Figure 11: Dialogues to handle sessions in CausalTrail. The dialogue on the left hand-side can be used to store a session, the dialogue on the right hand-side can be used to load a session. Sessions are stored in CausalTrail Session Files (cts).

A conditional probability table for the selected node pops up. Its name is given in the header of the window. Columns in the CPT with an italic label represent the parent nodes and their respective values. Values of the selected node are written in bold. 4.5 Queries

CausalTrail offers three ways to submit a query. A query can be

  1. entered directly, as in the console version of CausalTrail.
  2. entered interactively, by clicking on the nodes of the network and selecting their desired state.
  3. chosen from the Query History, containing all previously submitted queries of the current session.

The following sections cover these options in detail.
The result of query is shown in the Log and in the Query Control Panel. Submitted queries are listed in the Query History.

4.5.1 Direct Query Formulation

Queries can be entered directly using the Query Input widget. A query can be submitted by pressing Enter or by a click on the green tick. An example is shown in Figure 13.


Figure 13: Direct input of queries using the Query Input widget. Queries are submitted by a click on the green tick, or by pressing Enter. Clicking on the red cross resets the result box in the Query Control Panel.

Entering queries directly requires the user to be familiar with our query language. As this can not be expected from the general user, CausalTrail allows an interactive query construction, introduced in the next section.

4.5.2 Interactive Query Construction

Queries can be built interactively by the user. This permits the user to formulate queries without learning our query language. To start building a query, the user has to move the mouse over a node that should be a part of the query. A right click opens a context menu, as shown in Figure 14.


Figure 14: At the top, the context menu for nodes is shown. At the bottom, we show the context menu for edges. The context menus are empty if the network is not trained.

There are five options related to query construction:

  1. Probability of: Calculates the probability that this node obtains the specified value.
  2. Arg Max: Calculates the most likely value assignment for this node. It is not possible to combine (1) and (2).
  3. Condition on: Conditioning the probability of other nodes on the specified value of the currently selected one.
  4. Set Value to: Performs a do-intervention on the current node. The edges to its parents are deleted, and its own value is fixed to the selected one.
  5. Add Edge to: Adds an edge to another node. To add a new edge double click on the desired target node. The added edge will appear in red. If the new edge would induce a cycle, an error will be shown in the Log. Adding an edge triggers retraining of the entire network.

Once an item is selected, it appears in the Query Control Panel. Double clicking on an item in one the boxes in the Query Control Panel removes them from the current query. A colour code simplifies the identification of different query elements. A natural language wrapper around the query element boxes further facilitates the understanding of the query.
In addition to the operations on nodes, there is an operation on edges. A right click on an edge opens a context menu, as shown in Figure 14. Selecting Remove Edge removes the selected edge. Removed edges, are shown in grey. The style of the edge is changed from solid to dashed. As for adding an edge, removing one also triggers retraining of the network. An example containing all types of actions, except the Arg Max setting is shown in Figure 15.


Figure 15: An example query in the Student network. In this example, we compute the probability that Intelligence has the value i1, given the SAT has the value s1, Letter is set to l1, there is an additional edge from Intelligence to Letter, and the edge from Intelligence to Grade is removed. Conditions are shown in bright red, do interventions in yellow, query probabilities in green. Additional edges are red, removed edges are dashed and shown in grey.

A query can be calculated either by clicking on the green tick in the Query Control Panel, or by pressing Enter. To enter a new query, the button New Query has to be clicked.

4.5.3 Query History

Queries are stored in the Query History. There are two ways to reload a query contained in the history.

  1. A query can be directly selected via a double click in the Query History box, as shown in Figure 16.
  2. Using the yellow and green arrows in the Query Input and in the Query Control Panel, the user can shift through all previously submitted queries

The visualisation of the network and the Query Control Panel are adapted according to the selected query.


Figure 16: The Query History widget lists all valid, submitted queries. Double clicking a query reloads it. The network visualisation and the Query Control Panel are updated according to the query. In this example, the second query is selected.

4.6 Query Batch Files

To allow the user to quickly process a set of queries with networks trained on different data, CausalTrail offers Query Batch Files. A batch file can be created by clicking on Create Batchfile. Queries currently shown in the Query History are written to the batchfile. A batchfile can be executed by a click on Execute Batchfile. Results are shown in the Log.
